# Project & Kanban Module Roadmap

## Summary
This document defines a phased implementation roadmap for a small-business Jira-like module with:
- Projects
- Project members
- Task Kanban board
- Task comments

The goal is to execute phase-by-phase with clear entry/exit criteria and no mid-phase scope drift.

---

## Locked V1 Decisions
- Scope: Projects + Tasks
- Roles:
  - Company Admin / HR Manager: manage projects, members, tasks
  - Employee / Manager: view member projects, update status on assigned tasks, comment
- Kanban statuses (fixed): `TODO`, `IN_PROGRESS`, `REVIEW`, `DONE`
- Assignee model: single assignee per task
- Extras: comments only (no attachments)
- Visibility: employee sees only projects they are a member of
- Project lifecycle: `ACTIVE`, `COMPLETED`, `ARCHIVED`
- Completed/Archived projects: read-only until reactivated
- Member removal: blocked if member still has assigned tasks in that project
- Card ordering: no persisted order in V1 (status move only)

---

## Phase 0: Documentation Freeze
### Objective
Freeze endpoint, schema, permission, and test contracts before coding.

### Deliverables
- This roadmap document
- API contract section finalized
- Schema and constraints finalized
- Test matrix finalized

### Exit Criteria
- No unresolved design decisions
- Implementer can start coding without additional product decisions

---

## Phase 1: Prisma Schema + Migration
### Scope
- Add enums:
  - `ProjectStatus`
  - `ProjectTaskStatus`
  - `ProjectTaskPriority`
- Add models:
  - `Project`
  - `ProjectMember`
  - `ProjectTask`
  - `ProjectTaskComment`
- Add relations in `Company`, `Employee`, `User`
- Add indexes and constraints
- Create migration and generate Prisma client

### Exit Criteria
- Migration applies cleanly
- Prisma client generation succeeds

---

## Phase 2: Backend Project Core (Admin/HR)
### Scope
- New module: `backend/src/modules/project/`
- Admin/HR endpoints for project and member management

### Endpoints
- `POST /projects/admin`
- `GET /projects/admin`
- `GET /projects/admin/:projectId`
- `PATCH /projects/admin/:projectId`
- `PATCH /projects/admin/:projectId/status`
- `POST /projects/admin/:projectId/members`
- `DELETE /projects/admin/:projectId/members/:employeeId`

### Rules
- Company scope enforcement
- Duplicate member prevention
- Block member removal if assigned tasks exist

### Exit Criteria
- Project/member admin APIs pass functional tests

---

## Phase 3: Backend Tasks + Comments + Employee APIs
### Scope
- Task CRUD, task status updates, comment APIs
- Employee/manager scoped read + self-task status update + comments

### Admin/HR Endpoints
- `POST /projects/admin/:projectId/tasks`
- `GET /projects/admin/:projectId/tasks`
- `GET /projects/admin/tasks/:taskId`
- `PATCH /projects/admin/tasks/:taskId`
- `PATCH /projects/admin/tasks/:taskId/status`
- `GET /projects/admin/tasks/:taskId/comments`
- `POST /projects/admin/tasks/:taskId/comments`

### Employee/Manager Endpoints
- `GET /projects/me`
- `GET /projects/me/:projectId/tasks`
- `GET /projects/me/tasks/:taskId`
- `PATCH /projects/me/tasks/:taskId/status`
- `GET /projects/me/tasks/:taskId/comments`
- `POST /projects/me/tasks/:taskId/comments`

### Rules
- Employee/manager can update only assigned tasks
- Employee/manager can access only member projects
- Read-only enforcement for completed/archived projects

### Exit Criteria
- Full backend role/access matrix validated

---

## Phase 4: Frontend API + Reusable Components
### Scope
- Add API client: `frontend/src/lib/api/projects.ts`
- Build reusable components:
  - `ProjectKanbanBoard.tsx`
  - `ProjectFormModal.tsx`
  - `ProjectTaskFormModal.tsx`
  - `ProjectTaskDetailModal.tsx`
  - `ProjectMemberManager.tsx`

### UX
- Drag-drop for status change
- Fallback status action for mobile/non-drag interaction
- No persistent intra-column ordering

### Exit Criteria
- Components are typed and integrated with API layer

---

## Phase 5: Admin/HR Pages + Sidebar
### Scope
- Shared admin page and role route wrappers
- Add sidebar entry for companyadmin/hrmanager

### Pages
- `/dashboard/companyadmin/projects`
- `/dashboard/hrmanager/projects`

### Exit Criteria
- Admin/HR can manage end-to-end project/task flows from UI

---

## Phase 6: Employee/Manager Pages + Sidebar
### Scope
- Employee/manager projects page
- Add employee sidebar entry

### Page
- `/dashboard/employee/projects`

### Exit Criteria
- Employee can view own projects and update assigned task status + comments

---

## Phase 7: Validation + Release Gate
### Backend Checks
- Build passes
- Company scope and role checks validated
- Status/lifecycle guards validated

### Frontend Checks
- `npx tsc --noEmit` passes
- Kanban drag + fallback status actions validated
- Role-based visibility verified

### Release Gate
- All phase exit criteria met
- No unresolved TODO in project module

---

## API & Response Conventions
Use existing app convention:
- `{ message, data }`
- `{ message, data, meta }` for paginated lists

Pagination meta fields:
- `total`, `page`, `limit`, `totalPages`, `hasNextPage`, `hasPreviousPage`

---

## Out of Scope for V1
- Subtasks
- Attachments/files
- Sprints
- Time logging
- Story points
- Watchers/mentions/notifications
- Custom workflows per project
